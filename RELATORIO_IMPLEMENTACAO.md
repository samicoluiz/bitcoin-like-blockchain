# Relatório Técnico de Implementação Blockchain

## Descrição da Arquitetura
O sistema foi desenvolvido seguindo o modelo de rede entre pares onde não existe um servidor central coordenando as operações. Cada nó da rede funciona de maneira independente e assume as responsabilidades de cliente e servidor simultaneamente. A implementação em Python utiliza a biblioteca de sockets para estabelecer conexões diretas entre os computadores do laboratório. Internamente o nó é dividido em camadas que cuidam da validação de regras de negócio como saldo e valores positivos além da gestão de conexões de rede através de múltiplas threads. Esta estrutura permite que o nó receba novas transações e blocos enquanto realiza o trabalho pesado de mineração em segundo plano.

## Protocolo de Comunicação
A troca de informações entre os nós acontece por meio de mensagens serializadas em formato JSON que garantem a interoperabilidade do sistema. Para evitar problemas de leitura em fluxos contínuos de dados cada mensagem é precedida por um cabeçalho de quatro bytes que informa o tamanho total do conteúdo a seguir. O protocolo define cinco tipos principais de interação na rede. A mensagem NEW TRANSACTION é usada para propagar novos pagamentos criados por qualquer usuário. O comando NEW BLOCK serve para anunciar que um novo bloco válido foi encontrado e deve ser adicionado às cópias locais. Para a entrada de novos participantes existem as mensagens REQUEST CHAIN e RESPONSE CHAIN que realizam a transferência total da blockchain. Além disso o sistema possui mensagens de controle como DISCOVER PEERS e PEERS LIST que permitem aos nós descobrir outros endereços ativos na rede sem intervenção manual.

## Mecanismo de Consenso
O consenso entre os participantes é alcançado através do algoritmo Proof of Work simplificado com uma dificuldade estática. Um bloco só é considerado legítimo pela rede se o seu hash gerado pelo algoritmo SHA 256 começar com a sequência de caracteres 000. O processo de mineração exige que o nó altere continuamente um campo numérico chamado nonce até encontrar um resultado que satisfaça essa condição matemática. Quando um nó recebe um novo bloco ele executa uma série de verificações para garantir que o índice é sequencial e que o hash do bloco anterior corresponde ao que está gravado localmente. Em situações onde dois nós mineram blocos ao mesmo tempo o sistema resolve o conflito utilizando a regra da cadeia mais longa que estabelece a versão com maior quantidade de trabalho acumulado como a oficial.

## Limitações do Sistema
Uma das limitações atuais do projeto é a ausência de um sistema de persistência de dados em disco o que provoca a perda de todo o histórico se o processo for reiniciado. O sistema também não implementa criptografia de chave pública e privada para a assinatura de transações permitindo que qualquer usuário crie pagamentos em nome de outras contas se souber o endereço de origem. Outro ponto relevante é que a dificuldade de mineração não se ajusta automaticamente conforme o número de mineradores aumenta ou diminui na rede podendo tornar a criação de blocos muito rápida ou excessivamente lenta. A rede também é vulnerável a ataques de gasto duplo se um usuário mal-intencionado possuir mais poder de processamento do que o restante do grupo somado.

## Dificuldades Encontradas
A maior dificuldade técnica residiu em garantir o determinismo dos hashes entre diferentes máquinas e sistemas operacionais. Como o hash depende da representação textual do bloco qualquer mudança na ordem das chaves do dicionário JSON resultaria em um valor diferente inviabilizando a validação. Foi necessário forçar a ordenação das chaves durante a criação da string base para o cálculo. Outro desafio foi lidar com o recebimento de mensagens grandes através dos sockets pois os dados podem chegar fragmentados e exigem um controle rigoroso do buffer para não corromper o arquivo JSON. A sincronização de nós que entram na rede após a mineração de vários blocos também apresentou problemas iniciais de recursão infinita que foram resolvidos com a implementação de verificações de estado antes de solicitar novos dados.
